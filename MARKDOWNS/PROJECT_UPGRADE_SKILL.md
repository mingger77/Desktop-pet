# 项目改动规范（Project Upgrade Rules）

> 对项目进行任何修改时，必须严格遵守以下规范。

---

# 1. 基本原则

## 1.1 需求来源

1. 所有项目更新需求统一记录于 `UPGRADE.md`。
    
2. 未出现在 `UPGRADE.md` 中的需求不得擅自实现。
    
3. 不得根据模糊描述自行扩展需求。
    
4. 信息不足时必须主动提问。
    

---

## 1.2 分点定义

1. `UPGRADE.md` 一级列表中的每一项视为一个独立分点。
    
2. 每次仅允许处理一个分点。
    
3. 当前分点完成并经用户确认后，方可进入下一分点。
    

---

## 1.3 最小改动原则

1. 优先采用影响范围最小的方案。
    
2. 若存在以下两种方案：
    
    - 修改少量代码即可实现
        
    - 重构大量代码实现
        
    
    必须优先选择前者。
    
3. 未经用户明确要求，不得主动重构。
    

---

# 2. 修改前审计

正式修改前必须完成审计。

审计内容包括：

## 2.1 文件审计

明确：

- 涉及哪些文件
    
- 是否新增文件
    
- 是否删除文件
    

---

## 2.2 调用链审计

明确：

- 谁调用当前模块
    
- 当前模块调用谁
    
- 是否存在事件绑定
    
- 是否存在信号槽关联
    

---

## 2.3 数据流审计

明确：

- 输入来源
    
- 输出去向
    
- 配置读取位置
    
- 持久化存储位置
    

---

## 2.4 风险说明

说明：

- 可能影响哪些功能
    
- 是否影响兼容性
    
- 是否影响历史数据
    

---

未完成审计不得修改代码。

---

# 3. 修改难度评估

正式修改前必须评估难度。

从以下四个维度进行评级：

|维度|说明|
|---|---|
|上下文污染度|是否涉及多个文件|
|线程安全性|是否涉及线程、协程、信号槽|
|资产完整度|是否影响原有功能效果|
|架构侵入度|是否改变现有架构设计|

评级只能为：

- Low
    
- Medium
    
- High
    

示例：

- 上下文污染度：Low
    
- 线程安全性：Low
    
- 资产完整度：Medium
    
- 架构侵入度：Low
    

---

# 4. 方案判定规则

## 4.1 可直接修改

满足以下条件之一：

### 条件A

全部维度均为 Low

### 条件B

仅一个维度为 Medium

则允许直接修改。

---

## 4.2 必须先出方案

满足以下任一条件：

### 风险条件

- 涉及线程
    
- 涉及协程
    
- 涉及异步网络请求
    
- 涉及 PyQt 事件循环
    
- 涉及信号槽
    

### 文件条件

涉及两个及以上项目文件：

包括但不限于：

- py
    
- ui
    
- qss
    
- json
    
- yaml
    
- toml
    
- md
    

### 数据条件

涉及：

- 配置结构
    
- JSON账本
    
- 数据库存储
    
- 缓存结构
    
- 持久化格式
    

### 难度条件

- 任意一个维度为 High
    
- 两个及以上维度为 Medium
    

### 需求复杂度条件

单条需求包含多个并列动作：

例如：

- 且
    
- 并
    
- 以及
    
- 同时
    

出现两个及以上并列动作。

---

## 4.3 必须拆分需求

满足以下任一条件：

### 条件A

两个及以上维度为 High

### 条件B

单条需求包含四个及以上并列动作

则必须：

1. 拆分子任务
    
2. 重新审计
    
3. 重新评估风险
    

---

# 5. Git规范

## 5.1 修改前检查

开始新分点前必须检查：

```bash
git status
```

要求：

```text
working tree clean
```

否则不得继续。

---

## 5.2 记录回滚点

正式修改前必须记录：

```bash
git rev-parse HEAD
```

保存当前 Commit。

---

## 5.3 测试失败处理

若测试失败：

必须提供：

```bash
git restore .
```

或

```bash
git reset --hard <commit>
```

等回滚方案。

---

# 6. 修改过程规范

## 6.1 禁止脑补

禁止自行假设：

- 不存在的模块
    
- 不存在的接口
    
- 不存在的配置项
    
- 不存在的数据库结构
    
- 不存在的业务逻辑
    

发现信息不足时必须提问。

---

## 6.2 禁止扩大改动范围

不得因为实现方便而：

- 顺手重构
    
- 顺手优化
    
- 顺手清理代码
    

除非需求明确要求。

---

## 6.3 禁止越界操作

不得操作：

- 项目根目录之外路径
    
- workspace 沙盒之外路径
    
- 系统目录
    
- 注册表
    
- 用户敏感文件
    

---

# 7. 故障态机制

若首次修改后出现以下任一情况：

- 编译失败
    
- 单元测试失败
    
- 集成测试失败
    
- Nuitka构建失败
    
- PyQt事件死锁
    
- 程序无法启动
    
- 运行时异常
    

立即进入故障态。

---

进入故障态后：

禁止继续修改代码。

必须输出：

## 根因分析

说明故障原因。

## 影响范围

说明影响模块。

## 修复方案

给出修复思路。

## 回滚方案

提供回滚命令。

等待用户确认后方可继续。

---

# 8. 计划汇报模板

正式修改前必须输出：

## 审计结果

- 涉及文件
    
- 调用链
    
- 数据流
    

## 难度评估

- 上下文污染度：
    
- 线程安全性：
    
- 资产完整度：
    
- 架构侵入度：
    

## 实施计划

- 修改哪些文件
    
- 新增哪些依赖
    
- 是否影响现有事件
    
- 是否影响兼容性
    

## 测试方案

- 如何验证
    
- 如何回滚
    

---

# 9. 分点完成规则

每个分点完成后必须输出：

## 修改内容

说明实际改动。

## 测试结果

说明验证结果。

## 风险评估

说明剩余风险。

## Git状态

确认工作区状态。

---

输出完成后停止执行。

等待用户确认后才能进入下一分点。
